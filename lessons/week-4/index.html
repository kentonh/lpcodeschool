Exercises

Coin combinations

Have you noticed those coin slides used by modern cash registers that automatically give the correct amount of change? Build a website lets users enter an amount of cents and shows the least amount of coins needed to make that change. For bonus points, show more than one combination. Start with tests for small combinations and work up from there.

Word Order

Let's create a webpage that lets users input a block of text and returns a list of all the unique words, ordered by number of appearances.

For example, if the user inputs "hello world world", the page should show:

  • world
  • hello

For words that appear the same amount of times, order by which word appeared first.

Next Weekday

Make a webpage that allows users to input a day of the week (like Friday) and return the date of the next occurrence (like 7/11/2014). If a user enters the current day, it should return the date of a week from today - not today.

We can get the current date by writing var today = new Date. We can then get the current month using today.getMonth() or day with today.getDay(). There are many more ways you can get information about (and even change) dates. Check out the MDN documentation on Date.

Weekdays in Month

This function should take a weekday and a month as an argument ("Friday", "January") and return all the dates of that weekday in the given month. You can add this function to the previous webpage.

Exercise

Ready for something a little harder? Write a Pig Latin translator. Here are the rules of Pig Latin:

  • For words that start with a vowel, add "ay" to the end.
  • For words that start with one or more consonants, move all of the first consecutive consonants to the end and add "ay". (If the first consonants include "qu", move the "u" along with the "q". Don't forget about words like "squeal" where the "qu" doesn't come first!)
  • For words that start with "y", treat the "y" as a consonant.

Remember to break this down into small tests. When you get to consonants, don't try to solve it all at once. Instead, start with an example of a word that only has one consonant; then a word with two consonants; then a word with three; and then tackle "qu" and "y".

The .slice() method for strings may come in handy.

Once you have it working for one word, get it working for sentences.

And if you finish all of this, read about regular expressions and refactor your code to use them.

JavaScript BDD Evaluation

Starting here, at the end of every section, there will be an assessment for you to take. These assessments are designed to help you gauge if you're ready to move on to the next section.

These assessments will test that you understand the concepts from the section, and can you write good code using them - that simple. Programmers don't write new apps in isolation, and frequently reference code they've written and resources they've used; so reviewing your past code, and lessons from this and other sites, searching Google, and looking on Stack Overflow are all allowed. However, you should complete these assessments on your own, without asking for help from others (including asking a question on Stack Overflow), so that you can make sure you understand the concepts .

For this section, your assessment is to simply write a function in JavaScript, using BDD to guide your implementation. You can make up your own idea of something to build, or you can work on an exercise from the past section that you didn't get to earlier.

Please include a README file (simple text, explaining the purpose and background of the function)

When you finish, submit a link to your project on GitHub in your class Slack room. Your teachers will evaluate it based on the following criteria:

  • Does the function work?
  • Do you have complete spec coverage?
  • Do your specs have clear English descriptions?
  • Is your logic easy to understand?
  • Did you use descriptive variable names?
  • Does your code have proper indentation and spacing?
  • Did you include a README with a background information and your name?
  • Is the project tracked in Git, and did you regularly make commits with clear messages?

For each of these, your teachers will choose one of the following:

  • The code does not meet this standard.
  • The code meets this standard most of the time.
  • The code meets this standard all of the time.
  • The code exceeds this standard.

If your code meets all of the standards most of the time or better, then you are ready to move to the next section! If not, you should spend some more time practicing this section, and then submit another evaluation.

I'd strongly recommend you grade yourself before turning in your assessment. Better yet, wait at least an hour after you finish, then come back and grade yourself. You'll catch many things you didn't notice as you were coding.
