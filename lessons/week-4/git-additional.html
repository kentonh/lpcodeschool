Git Additional Reading


Making commits

It's not always clear when to make a commit. To start, I'd suggest committing every time you finish the "refactor" step after making a spec pass.

Another good rule of thumb is that you should never have to write the word "and" in a commit message. If you do, it should probably have been two commits.

The best description I've heard of when to make commits is that they should represent a "logically separate changeset".

Remember to make commits in the present tense, like this:

add cookie eating feature

not like this:

adds cookie eating feature

Another way to think of it is that the message should answer the question and complete the sentence, "What will this commit do when applied? It will... ".

READMEs

Treehouse doesn't give you much information on what to put in your README files. Here are a few things that you should include in every README:

  • Setup instructions. This is less important when working with flat HTML/CSS/JS files like we are now, but you should still include something instructing the user to clone the repository and then open the HTML file in the browser.
  • Copyright and licensing information. The most common licenses for open source projects are the MIT license, which lets users use your code in any way, and the GPL v2, which requires your users to open source any changes they make to your code or any projects which use your code. To keep things simple, you can simply put the name of the license in your README, rather than the full text.
  • Names of the authors and optionally contact information
  • Known bugs

Here are a couple examples of great READMEs: simple_form and capybara-webkit.

Configuring your own computer

The lesson mentions that you should set your name and email on your computer using git config --global. There are two other configurations I'd suggest you make on your computer. First, tell Git to use color in its output:

$ git config --global color.ui true

Next, you might notice that there are some files that are not part of your project; for example, Macs have hidden files in most directories called .DS_Store, and Windows directories often contain files called Thumbs.db. To ignore these files, create a file in your home directory called .gitignore_global. Now you'll need to add some rules of what types of files to ignore. Github has a great example that you can copy into your file. Finally, we need to tell Git to use this as a rule for all of the projects you set up on this computer:

git config --global core.excludesfile ~/.gitignore_global

Finally, to make Git use Sublime instead of vim when it needs to interact with you, run $ echo "export EDITOR='subl -w'" >> ~/.bash_profile.

Git prefers that you add an empty line to the end of every file (it's got something to do with cross-operating system compatibility). Rather than having to remember to do so on every file you create, you can set up Sublime to do this for you. Click the Sublime Text 2 menu, Preferences, Settings - User, and then add this to the settings:

"ensure_newline_at_eof_on_save": true

Git (and most programmers) also doesn't like it when you have trailing spaces at the end of your lines. There is a preference to remove those in Sublime, too:

"trim_trailing_white_space_on_save": true

It's also very standard to use two spaces instead of tabs for indenting - different computers show tabs as taking different amount of room, but they all represent spaces the same way. Add these preferences in Sublime:

"tab_size": 2,
                    "translate_tabs_to_spaces": true
